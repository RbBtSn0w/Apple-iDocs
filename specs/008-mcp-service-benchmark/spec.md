## Clarifications

### Session 2026-03-19

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 项目内完成四个服务的可访问验证 (Priority: P1)

作为项目维护者，我希望在当前仓库范围内同时接入 `idocs` CLI、`apple-docs-mcp`、`apple-doc-mcp` 和 `sosumi.ai`，以便在不修改全局设置的前提下验证四个服务是否都能被发现、调用和区分。

**Why this priority**: 如果四个目标不能在同一项目环境下稳定访问，后续任何能力、性能或准确性对比都没有基础。

**Independent Test**: 在全新本地环境中，仅使用仓库内配置和说明完成四个目标的接入，然后逐一执行最小可用请求，确认都能返回可识别结果。

**Acceptance Scenarios**:

1. **Given** 项目内已提供完整的本地接入说明与配置, **When** 维护者按说明初始化测试环境, **Then** 四个目标都可在不依赖全局配置变更的情况下被访问
2. **Given** 四个目标已注册到同一测试环境, **When** 维护者分别执行最小验证请求, **Then** 每个目标都能返回可识别的结果或明确可诊断的失败信息

---

### User Story 2 - 使用统一测试数据执行四路能力测试 (Priority: P1)

作为评估者，我希望用一套统一整理的测试数据访问四个目标，以便比较它们在相同任务下的返回内容、响应耗时和完成情况。

**Why this priority**: 统一输入是公平比较的前提，否则不同服务的结果没有可比性。

**Independent Test**: 使用预先整理的测试集，对四个目标执行共享能力任务，并单列执行各自扩展能力任务，最终生成逐条记录的事实表和评分表。

**Acceptance Scenarios**:

1. **Given** 已定义统一测试集, **When** 评估者对四个目标运行共享能力任务, **Then** 每个测试任务都会生成按目标分列的结果记录并参与基础总分计算
2. **Given** 某个目标具备其他目标不具备的专属能力, **When** 评估者执行扩展能力任务, **Then** 结果会单列展示且不影响基础总分
3. **Given** 某个目标无法完成某条任务, **When** 测试继续执行, **Then** 结果记录中会保留失败原因、是否可重试和该条对比的可用性说明

---

### User Story 3 - 形成面向 AI 使用场景的对比结论 (Priority: P2)

作为需要为 AI 选择文档来源的使用者，我希望看到一份有结论的对比结果，明确四个目标在内容准确性、响应速度、内容完整性、成本可观测性和稳定性上的差异。

**Why this priority**: 仅有原始测试结果不足以指导选型，必须给出可执行的结论和适用场景建议。

**Independent Test**: 阅读最终对比结果后，不查看底层实现也能区分事实记录与加权评分，并判断四个目标分别适合哪些使用场景，以及它们的主要短板。

**Acceptance Scenarios**:

1. **Given** 已完成四路测试并收集原始记录, **When** 评估者查看汇总结果, **Then** 可以同时看到客观事实记录、加权评分结果、评分依据和适用场景结论
2. **Given** 某些目标无法直接提供成本信息, **When** 汇总结果生成, **Then** 报告会明确标记该指标不可直接观测，并使用统一的替代说明避免误导性比较
3. **Given** 某个结论被标记为更准确、更快或更稳定, **When** 审阅者追溯其来源, **Then** 可以找到对应的官方证据、运行记录和统计依据

---

### User Story 4 - 支持后续重复复测 (Priority: P3)

作为长期维护者，我希望这套验证环境和测试数据可以重复执行，以便在服务升级、模型调整或数据源变化后快速复测并追踪差异。

**Why this priority**: 外部文档源和服务行为可能随时间变化，一次性验证无法支撑持续判断。

**Independent Test**: 在后续任意一次复测中，维护者能够复用同一套环境说明、测试数据和对比维度，完成新一轮记录并与历史结果比对。

**Acceptance Scenarios**:

1. **Given** 已存在一轮完整测试记录, **When** 维护者再次执行同一套验证流程, **Then** 新结果仍能按相同字段与上一次结果进行比较
2. **Given** 某个外部目标的行为发生变化, **When** 维护者进行复测, **Then** 差异会在结果记录中被明确标示而不是覆盖旧结论

### Edge Cases

- 某个外部目标暂时不可用、限流或需要更长冷启动时间时，测试结果如何区分“接入失败”“临时不可用”和“能力不足”？
- 某些目标无法提供 token 消耗或等价成本数据时，如何避免把“不可观测”误判为“低成本”？
- 四个目标返回的内容结构差异很大时，如何保证准确性与完整性的评价口径仍然一致？
- 同一查询在不同时间返回不同结果时，如何识别是数据源更新、缓存影响还是服务不稳定？
- 某个目标仅在专属能力任务中可用时，如何避免其扩展优势污染共享能力基础总分？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在当前项目范围内提供 `idocs` CLI、`apple-docs-mcp`、`apple-doc-mcp` 和 `sosumi.ai` 四个目标的本地接入方式
- **FR-002**: 系统 MUST 将三项外部服务的接入限定在项目级配置中，不要求也不依赖用户修改全局 MCP 设置
- **FR-003**: 系统 MUST 为四个目标定义统一的最小可用验证步骤，以确认每个目标已成功接入且可被独立访问
- **FR-004**: 系统 MUST 准备至少 12 条可复用测试任务，并覆盖至少以下任务类型：精确搜索、模糊搜索、文档抓取、主题理解、失败处理、长内容或复杂内容评估
- **FR-005**: 系统 MUST 将测试任务分为共享能力任务和扩展能力任务，其中共享能力任务用于计算基础总分，扩展能力任务单列展示且不计入基础总分
- **FR-006**: 系统 MUST 使用同一套输入文本或路径对四个目标分别执行访问和测试，并为每条测试保留原始输出记录或摘要证据
- **FR-007**: 系统 MUST 为每次测试记录统一的客观字段，至少包括 `Run ID`、目标名称、任务标识、尝试次数、执行状态、开始时间、完成时间、总耗时、调用次数、输出长度、估算 token、token 是否可观测、错误分类、证据引用、准确性结论、完整性结论和审阅备注
- **FR-008**: 系统 MUST 在可直接获取时记录 token 消耗或等价成本信息；在不可直接获取时明确标记为“不可观测”，不得留空或按低成本处理
- **FR-009**: 系统 MUST 对四个目标的返回内容进行准确性评估，并基于 Apple 官方文档页面或官方 DocC 数据等预先定义的参考来源给出可解释结论
- **FR-010**: 系统 MUST 对四个目标的返回内容进行完整性和可用性评估，说明结果是否足以支持 AI 完成用户任务
- **FR-011**: 系统 MUST 使用固定的 100 分制评分模型，权重为：准确性 35、完整性 20、效率 15、Token 或上下文成本 10、稳定性 10、可诊断性 10
- **FR-012**: 系统 MUST 将客观事实记录与加权评分分开呈现，不得以主观描述替代原始事实记录
- **FR-013**: 系统 MUST 将执行速度、内容准确性、内容完整性、稳定性、错误可诊断性和成本可观测性纳入最终对比维度
- **FR-014**: 系统 MUST 将同一任务执行 3 次，并区分 1 次冷启动和 2 次热启动记录，以支撑速度和稳定性结论
- **FR-015**: 系统 MUST 对所有“更准确”“更快”“更稳定”“更省 token”类结论提供可追溯依据，分别指向官方证据、运行记录或统计结果，并说明是实测值还是估算值
- **FR-016**: 系统 MUST 产出可阅读的汇总结果，明确四个目标各自的优势、限制、适用场景和不推荐场景，并回答日常查询、深度阅读、低成本定位、故障诊断四类选型问题
- **FR-017**: 系统 MUST 保证同一套验证流程可以重复执行，并能将新一轮结果与历史结果并列比较
- **FR-018**: 系统 MUST 明确标注测试前提条件，包括网络依赖、账号或密钥前提、缓存状态、以及可能影响结果的环境差异
- **FR-019**: 系统 MUST 在某个目标测试失败时继续执行其他目标和其他测试项，除非整体测试环境已无法产出有效对比
- **FR-020**: 系统 MUST 在任务对某个目标天然不适用时标记为 `Not Applicable` 并说明原因；只有该任务属于共享能力范围时，缺失能力才可计入基础评分影响

### Key Entities *(include if feature involves data)*

- **测试目标 (Service Target)**: 被纳入比较的单个服务实例，包括 `idocs` CLI 和三个外部文档服务
- **测试任务 (Test Scenario)**: 以统一输入表达的单个验证任务，如搜索、取文档、理解主题或处理异常请求
- **测试运行 (Evaluation Run)**: 某一时间点基于固定环境对全部测试目标执行的一轮完整验证
- **共享能力任务 (Shared-Capability Task)**: 四个目标都应具备的共通能力测试任务，用于计算基础总分
- **扩展能力任务 (Extended-Capability Task)**: 仅部分目标具备的专属能力测试任务，用于单列展示扩展优势，不计入基础总分
- **评估指标 (Evaluation Metric)**: 用于比较四个目标表现的标准字段，如速度、准确性、完整性、稳定性和成本可观测性
- **参考答案 (Reference Evidence)**: 用于判断返回内容是否正确、充分和可用的基准资料
- **客观记录 (Evidence Record)**: 每次运行保留的原始事实数据，用于支撑评分、结论和后续复核
- **对比报告 (Comparison Report)**: 汇总单轮或多轮测试结果、差异说明和适用场景结论的输出物

## Assumptions

- 本次功能范围仅包含一个仓库内的 `idocs` CLI 和三个指定外部服务，不扩展到其他文档源或通用搜索服务。
- “项目级接入”指维护者只需在当前仓库内完成配置、说明和验证，不需要修改任何系统级或账号级全局配置。
- 基础总分只覆盖四个目标都应具备的共享能力；专属能力通过扩展能力任务单列展示，不参与基础总分比较。
- token 消耗只有在目标直接暴露相关信息时才纳入实测数值对比；若目标不暴露，则以“不可观测”标记并补充解释。
- 准确性判断以 Apple 官方文档页面或官方 DocC 数据等权威资料为参考，不以某个服务自身返回内容作为唯一判定依据。
- 每条任务默认执行 3 次，其中 1 次视为冷启动，2 次视为热启动，用于区分偶发波动与稳定表现。
- 对比结论服务于“帮助 AI 选择更合适的文档来源”这一目标，而不是对外发布通用行业排名。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 四个目标均可在同一项目级测试环境中完成接入验证，且全过程不要求修改全局 MCP 设置
- **SC-002**: 至少准备 12 条可复用测试任务，其中至少 8 条具备可核对的官方真值，至少 2 条验证错误处理真实性，至少 2 条验证长内容或复杂内容下的完整性差异
- **SC-003**: 100% 的测试记录包含统一客观字段：目标名称、任务标识、尝试次数、执行状态、总耗时、调用次数、内容长度、token 可观测性和失败原因或成功证据
- **SC-004**: 所有共享能力任务都使用完全相同的输入文本或路径执行，且基础总分仅由共享能力任务生成
- **SC-005**: 对所有成功返回结果中的至少 90%，评估者能够基于参考答案给出明确的准确性和完整性判断，而不是“无法判断”
- **SC-006**: 最终汇总结果对四个目标都给出至少 1 个推荐场景和 1 个限制场景，并能明确回答日常查询、深度阅读、低成本定位、故障诊断四个选型问题
- **SC-007**: 在相同测试数据下完成第二轮复测时，维护者能够沿用同一比较字段输出并识别与首轮结果的差异
